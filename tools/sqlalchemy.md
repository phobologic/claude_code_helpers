# SQLAlchemy Async Patterns & Alembic Conventions

## SQLAlchemy Async Patterns

### Relationship Loading

- **Never access unloaded ORM relationship attributes outside an async session context.**
  Doing so triggers a synchronous lazy load that fails with `MissingGreenlet`.
- Use `selectinload()` to eagerly load relationships, or use `inspect(obj).attrs.<rel>.loaded`
  to guard before access.
- Capture object IDs into plain integers *before* calling anything that internally calls
  `expire_all()` — accessing `.id` on an expired object triggers a synchronous lazy load
  that fails with `MissingGreenlet`.

### Session Configuration

- **Set `expire_on_commit=False`** on sessions used in background tasks. Default
  `expire_on_commit=True` invalidates all attributes after commit, causing MissingGreenlet
  errors if attributes are accessed later outside a session context.
- **Avoid `expire_all()` in async contexts.** It marks all loaded objects as expired across
  greenlet boundaries, causing MissingGreenlet on next attribute access.
- When you need fresh state after an external write, prefer closing and reopening the session
  over calling `expire_all()`.

### State Synchronization

- **Call `await session.refresh(obj)` or `await session.expire(obj)`** (not `expire_all()`)
  to refresh specific objects when you know state may have changed.
- Before reading state that an HTTP request may have changed, reload the specific object
  rather than expiring the entire session.

### Test Isolation

- **Scope exact-count queries by a stable discriminator** (e.g., `game_id`, `user_id`) when
  using `StaticPool` in tests. A shared in-memory connection exposes rows from parallel or
  sequential test cases — unscoped `COUNT(*)` queries will see them all.

### Coverage Measurement

- **Always set `concurrency = ["greenlet"]`** in `[tool.coverage.run]` (pyproject.toml).
  SQLAlchemy's async engine uses greenlet internally; without this setting, coverage.py
  silently misses all code executed inside async DB sessions. This produces dramatically
  misleading coverage numbers for any module that touches the ORM.

## Alembic Migration Conventions

### Before Writing a Migration

```bash
uv run alembic heads   # Must show exactly one head — if not, resolve the branch first
uv run alembic current # Confirm which revision the local DB is at
```

Always verify the current head before writing a new migration. Use the printed hash as
`down_revision` in your migration file — never copy it from an existing file without
checking.

### Generating Migrations

```bash
# Autogenerate from model changes (preferred):
uv run alembic revision --autogenerate -m "describe the change"

# Manual stub for a migration that can't be autogenerated:
uv run alembic revision -m "describe the change"
```

**Never invent a revision ID** — always let Alembic generate it. Hand-written IDs create
collision risk and make the history harder to audit.

### Timestamps and Server Defaults

- When adding `server_default` for timestamps, verify the column works within the async
  session context before committing the migration.
- Prefer `server_default=func.now()` over Python-side `default=datetime.utcnow` for
  columns that should be set by the database.

### Applying Migrations

```bash
# Local development:
uv run alembic upgrade head

# Against a Railway-hosted database (adapt runner to your project):
railway run uv run alembic upgrade head
```

Always apply migrations before deploying code that depends on the new schema.
